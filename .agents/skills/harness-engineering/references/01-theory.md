# 01 - 理论根基（公理 / 误区 / 隐喻 / 角色跃迁）

> 本文回答："为什么需要 Harness？" 用 3 条公理 + 5 个误区破除 + 1 个隐喻建立直觉。

---

## 1. 核心隐喻：野马与缰绳

> **AI Agent = SOTA 模型（野马） + Harness（缰绳 / 驾驭系统） = 千里马**
> —— Mitchell Hashimoto，因 TRAE.ai 引入中文工程圈

### 隐喻的关键

Harness Engineering **不是**：

- 更好的提示词（Prompt Engineering ≠ Harness）
- 更强的模型（Foundation Model ≠ Harness）
- 更多的 Memory（Memory ≠ Harness）
- 更复杂的 Agent 自主性（Autonomy ≠ Harness）

Harness Engineering **是**：

- **"优化模型运行的环境与机制"**
- 不去改变马的基因（模型本身），而是为它设计专业马具
- 当模型遇到问题时，**通过一套工程化机制从根本上避免同类问题再发生**

### 与 Prompt/Context Engineering 的关系

```
        Prompt Engineering ⊂ Context Engineering ⊂ Harness Engineering
                (1 个对话)        (整个 session)       (整套系统)
```

- Prompt Engineering：单次提问技巧
- Context Engineering：管理 LLM 的上下文窗口
- **Harness Engineering：除 LLM 本身外，让 Agent 能干活的一切基础设施**

## 2. 三条公理（第一性原理推导起点）

### 公理 1：意图转化链有信息损耗

**论述**：软件工程的本质 = 把人脑中的模糊意图，经

```
需求 → 设计 → 编码 → 测试
```

逐步精确化为可执行代码。**每一步都有损耗**。AI 改变了各环节的损耗率，但**不消除链本身**。

**推论**：

- **源头的损耗传播最远**。需求阶段错 1%，到代码可能错 10%
- 因此 AI 应**贯穿全链条**降低损耗，不只是编码段提效
- 仅靠"AI 写得快"是局部最优，**全链信噪比**才是全局最优

### 公理 2：LLM 本质三特征

| 特征                   | 表现                          | 工程后果                              |
| ---------------------- | ----------------------------- | ------------------------------------- |
| **输出由上下文决定**   | 对私有知识掌握为零            | 必须显式注入项目上下文                |
| **输出是概率性的**     | 同样输入可能不同输出          | 单步成功率有上限 → **多步连乘必失稳** |
| **工作记忆有限且易失** | 超长上下文 Lost in the Middle | **不能塞百科全书，要用索引**          |

**关键推论**（来自公理 2）：

- 多步任务 = 错误指数累积
  - 如果每步 95% 成功率，10 步 = 0.95^10 ≈ 60% 成功率
  - 20 步 ≈ 36%
- **解决方案**：约束密集、步长短、多层验证
  - 这是为什么"小任务 + 多层验证"是核心实践

### 公理 3：人类认知是稀缺资源

**论述**：AI 时代的瓶颈是 **人的审查和决策带宽上限**。

- AI 把"看全代码"门槛打到零（穷举扫描成本 ≈ 0）
- **但**人的注意力没有变多
- 因此最优策略是：**优化工程师认知带宽的分配**

**人 vs AI 的新分工**：

- **AI 的优势**：穷举、扫描、模式识别、海量上下文检索
- **人的优势**：判断什么重要、设计决策、责任承担
- **错误分工**：让 AI 决策 + 人逐行 review → 双方都不擅长
- **正确分工**：让 AI 穷举/扫描 + 人聚焦判断 → 各扬其长

> **"经验的价值正在从'能看全'转移到'能判断什么重要'。"** —— 美团

## 3. 五个必须破除的误区

每个误区都是工程师/团队在 AI Coding 时常见的"自我合理化"，破除它们是建立正确 Harness 思维的前提。

### 误区 1："代码喂给 AI 就能理解项目"

**反例**：把整仓 50 万行代码塞给 AI，问"这个 bug 在哪"。

- 结果：AI 抓不到重点，回答模糊
- **原因**：信噪比太低 + 关键关系不显式

**真相**：

- **信噪比 + 结构化程度** 决定上下文价值，**不是代码量**
- 1000 行结构化 SPEC > 10 万行混乱代码
- **dev-map 是索引，不是百科全书**

### 误区 2："AI 不适合做复杂系统"

**反例论调**："我们这种复杂系统 AI 写不了，老老实实手写吧。"

**真相**：

- 不是 AI 不够聪明，是**关键知识没被结构化提供给它**
- 复杂系统的领域知识是工程师脑中的隐性知识
- **正确做法**：把"老员工才懂的潜规则"显式化为 Rule / Skill / dev-map
- **结果**：JK Launcher 全部代码 AI 写、美团 31 万行 90% AI 写，都是工程化做对了的复杂系统

### 误区 3："AI 提效 = 更快写代码"

**反例**：买 Cursor 但只用 Tab 补全 → 局部提效，全链条没变

**真相**：

- 公理 1 说源头损耗传播最远
- 真正的 AI 提效是**全链条**：
  - **需求**：AI 引导用户结构化表达
  - **设计**：AI 提多方案 + 权衡
  - **编码**：AI 执行
  - **验证**：AI 帮人审查、生成测试用例

### 误区 4："通过测试就能提交"

**反例**：AI 写了代码，所有单测过了，PR 提交 → 上线后规范一片混乱

**真相**：

- 测试只验**行为层**（这个函数返回什么）
- 测试**无法**捕获：
  - 设计偏差（架构不对但行为对）
  - 规范违反（命名/分层/依赖方向错了）
  - 偷删测试（覆盖率掉了但单测过）
- **解决方案**：Scripts 总验证脚本（**多层门禁**：编译 + 测试 + 静态规范 + 测试数量对比）

### 误区 5："AI 应像人一样独立完成整个任务"

**反例**：让一个 Agent 从需求理解到代码提交全自主完成 → 失控

**真相**：

- 公理 2 推论：多步连乘必失稳
- **全自主不可靠**，无论模型多强
- **解决方案**：
  - 小任务 + 多层验证（约束密集，步长短）
  - 多 Agent 结构化调度（每个 Agent 一段，互不越界）
  - **关键纪律**：单 Agent 失稳时再拆，**不是一开始就堆 7 个**

> **"AI 不是助手，AI 更像是一支执行力极强但必须被制度化管理的团队。"**

## 4. 角色跃迁：人在 AI 时代的新定位

### 跃迁路径

```
Phase 1: 写代码工人 (Code Writer)
              │
              ▼ AI 进入工作流
Phase 2: 用 AI 加速的代码工人 (AI-Augmented Writer)
              │
              ▼ 建立局部规范
Phase 3: 建立 Skill 的 AI 用户 (Skill Designer)
              │
              ▼ 整体 Harness 思维
Phase 4: 完整 Harness 系统设计师 (Harness Engineer)
```

### 跃迁的核心标志

> **"我们不再是逐行编码的工人，而是设计蓝图、定义规则、验收最终成果的架构师。"** —— TRAE.ai

> **"当 90% 代码由 AI 生成，团队成员的工作重心应从'写代码'转向'设计并维护一个能让 AI 可靠产出代码的工程环境'。"** —— 美团

### 各 Phase 的关键特征

| Phase | 关键行为                                         | 主要工具                                         | 痛点         |
| ----- | ------------------------------------------------ | ------------------------------------------------ | ------------ |
| 1     | 手写代码                                         | IDE                                              | 慢           |
| 2     | AI Tab 补全 + Chat                               | Copilot/Cursor 基础功能                          | 质量参差不齐 |
| 3     | 写项目 Rule、定义 Skill、用结构化 prompt         | Cursor Rules / .cursorrules / Claude Code Skills | 协作时不一致 |
| 4     | 设计多 Agent 流程、Scripts 门禁、dev-map、Pre-PR | 完整 Harness 栈                                  | 持续演进维护 |

---

## 5. OpenAI 引爆案例（角色跃迁的实证）

> 一个仅三人小团队，几乎不手写代码，通过引导 AI Agent，**5 个月构建百万行代码级别复杂产品**，累计合并约 1500 个 PR。

**这印证了**：

- 角色跃迁是真实可达成的
- 关键在于 **Harness 建设深度**，不在于团队规模
- "工程师从执行者到设计者的角色跃迁"

## 6. 为什么本 Skill 存在

**回到第一性原理**：

如果你信公理 1（意图损耗链）、公理 2（LLM 三特征）、公理 3（人脑稀缺），

那么你**必然**得出：

- 需要 Context Engineering（→ 公理 1）
- 需要硬约束门禁（→ 公理 2）
- 需要人机分工优化（→ 公理 3）
- 需要 Knowledge as Code（→ 公理 1 + 2）
- 需要 Error-Driven 闭环（→ 公理 2）
- 需要结构化多 Agent（→ 公理 2）

**这就是 Harness Engineering 的 6 条核心实践**——它们不是被发明的，是被推导出来的。
