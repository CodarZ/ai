## 第一部分：反模式总览（30+ 条）

### 1. 概念误用类

| 反模式                    | 真相                                        |
| ------------------------- | ------------------------------------------- |
| "Harness = 更好的 prompt" | Harness 是环境与机制，不是 prompt 技巧      |
| "Memory 是核心组件"       | 团队级 Harness 里**对话级 Memory** 应靠边站 |
| "Rule 能解决所有约束"     | Rule 是软约束，AI 会**解释性执行**          |
| "把整仓喂给 AI 它就懂"    | 信噪比决定上下文价值，不是代码越多越好      |
| "AI 不适合复杂系统"       | 不是 AI 不够聪明，是关键知识没结构化        |
| "AI 提效 = 更快写代码"    | 全链条损耗才是关键，编码段只是一段          |

### 2. 流程错误类

| 反模式                                  | 真相                           |
| --------------------------------------- | ------------------------------ |
| 下游自己改上游文档                      | 必须打回上游修改，保持责任边界 |
| PM Agent 越界做专业判断                 | PM 只做路由                    |
| 一个 Agent 干到底                       | 必然失稳                       |
| 去中心化自由协商 (AutoGen GroupChat 类) | 路径不稳定、责任不清、难维护   |
| "动态招聘"型多 Agent                    | 角色边界漂移、维护成本高       |
| 流程靠 PM 长文记忆                      | 必须抽成可校验资产             |
| 代码审查与测试验证合并                  | 两道独立的关不能并             |
| 让最强模型干所有 Agent 的活             | 模型应分层配置                 |

### 3. 验证缺失类

| 反模式                      | 真相                          |
| --------------------------- | ----------------------------- |
| "通过测试就能提交"          | 测试只验行为，不验设计偏差    |
| "我觉得做完了"              | 必须 Scripts 判定通过         |
| 完成幻觉（流程跑完 = 做对） | 必须事后验证                  |
| AI 自动生成全部测试用例     | 漏隐性高危、堆无价值边缘 case |
| "这错误本来就有"            | 基线对比堵住                  |
| 单模型审单模型              | 应高阶审低阶 + 跨厂商对抗     |
| Pre-PR 缺失，CR 找错字      | 木桶效应吞掉 AI 提效红利      |

### 4. 知识管理类

| 反模式                            | 真相                   |
| --------------------------------- | ---------------------- |
| 项目级知识库做百科全书            | 应该是**索引**         |
| 个人偏好上升为团队规范            | 应纳入仓库资产         |
| 规范停留在文档不进工具链          | **一纸空文**           |
| 同一规则散落 Rule / Memory / 聊天 | 三句话定位法明确归属   |
| dev-map 由 PM 维护                | 应由**改代码的人**维护 |
| 知识在 Confluence/Notion          | 与代码漂移             |
| 知识口口相传                      | 新人补不上             |

### 5. 起步策略类

| 反模式                           | 真相                     |
| -------------------------------- | ------------------------ |
| 第一天就把所有 Rule/Agent 全搭齐 | 应渐进演化               |
| 推倒重来式重构                   | 见缝插针                 |
| 申请专项排期重构                 | 拆解能力决定一切         |
| 试图人工遍历大代码库             | 专家定向 + AI 穷举       |
| 代码库混乱时直接放 AI 写         | AI 不纠偏反而放大差异    |
| 配 AI Rule 就万事大吉            | 真正瓶颈在人的共识       |
| 跳过"人人对齐"直接配 Rule        | 顺序错了 Rule 是一纸空文 |
| 让每人独立摸索 SOP               | 主 R 打样后分发          |

### 6. 状态管理类

| 反模式                                   | 真相                         |
| ---------------------------------------- | ---------------------------- |
| Prompt Engineering 让 LLM 自维护复杂状态 | LLM 必须视为无状态 CPU       |
| 长上下文塞所有信息                       | Lost in the Middle           |
| 信息注入位置随意                         | 注入边界（前/中/后）至关重要 |

### 7. AI 解释性执行（最隐蔽）

| 反模式      | AI 的借口            |
| ----------- | -------------------- |
| 跳过 Rule   | "这次特殊"           |
| 修改方案    | "历史遗留"           |
| 不写测试    | "等价验证"           |
| 直接 commit | "明显是对的"         |
| 删测试      | "这测试本来就有问题" |
| 改 baseline | "这报警本来就有"     |

→ 对策：能脚本化的尽量脚本化，让 Scripts 替代 Rule 的硬约束。

## 第二部分：5 处冲突裁决

合并 4 篇文章时发现的潜在冲突。这是本 skill 的**核心增量价值**——不是 4 篇文章的并列，而是裁决过的合并。

### 冲突 1：Memory 的地位

| 文章                 | 立场                                           |
| -------------------- | ---------------------------------------------- |
| 腾讯落地文（白家杰） | "团队级 Harness 里 Memory 应**靠边站**"        |
| TRAE 文              | "**三层记忆架构**（短期/长期/知识检索）必须有" |

**裁决**：**分场景。两人说的不是同一个 Memory**。

- 腾讯反对的是 **对话级 Memory**：
  - Claude/ChatGPT 自动总结的会话记忆
  - 个人口头约定
  - 散落在聊天里的"上次说要 A"
- TRAE 描述的是 **系统级 Memory**：
  - 外部存储的 RAG 知识库
  - 向量数据库
  - 持久化的状态存储

→ **前者要靠边，后者必须有**。

讨论 Memory 时，**先问"是哪种"**。

### 冲突 2：AI 自主程度

| 文章                 | 立场                                         |
| -------------------- | -------------------------------------------- |
| 第一性原理文（腾讯） | "AI **全自主不可靠**"                        |
| 腾讯落地文（白家杰） | "**全部代码由 AI 编写**"（JK Launcher 实证） |

**裁决**：表面冲突，实则一致。

- "全部代码由 AI 编写" 是在 **Harness 严格约束下**
  - 有 Rule、Skill、Scripts、多 Agent、Workflow 包围
  - AI 在每个小步骤受约束
- "AI 全自主不可靠" 反对的是 **无约束的全自主**
  - 让 AI 自由发挥从需求到上线
  - 没有验证门禁

→ 关键差异：**是否有 Harness 包围**。

→ 推论：批判"AI 自主"时要看是否有 Harness。

### 冲突 3：方法论起点

| 文章                 | 起点                       |
| -------------------- | -------------------------- |
| 美团文               | 先**人人对齐**，再人机对齐 |
| 第一性原理文（腾讯） | 从**公理**出发演绎推导     |
| 腾讯落地文（白家杰） | 从**最痛的问题**开始       |

**裁决**：互不矛盾，**三层关系**。

```
理论层: 公理（第一性原理文）
   ↓
组织层: 人人对齐（美团文）
   ↓
落地层: 从痛点切入（腾讯落地文）
```

- **想理解为什么** → 从公理出发
- **想团队落地** → 先做人人对齐
- **想立刻动手** → 从最痛的问题开始

→ 三者是**不同层级的回答**，不是不同方案。

### 冲突 4：Sub-Agent 数量

| 文章       | 推荐                                  |
| ---------- | ------------------------------------- |
| 腾讯落地文 | **7 个 Agent**（PM + 6 个专业 Agent） |
| TRAE 文    | 默认 **Plan-and-Execute**，按需叠加   |

**裁决**：**通用原则与具体落地**。

- TRAE 是 **通用建议**：渐进引入
  - 默认单 Agent
  - 失稳了再拆
- 腾讯是 **大型团队工程的具体落地**：
  - JK Launcher 这种规模演进到 7 个

→ **本质一致**：单 Agent 失稳再拆，不是一开始就堆 7 个。

→ 推论：小项目用 1 个，中型用 2-3 个，大型才用 7 个。

### 冲突 5：CR 的价值

| 文章       | 强调                               |
| ---------- | ---------------------------------- |
| 美团文     | 人工 CR 聚焦**"是否解决正确问题"** |
| 腾讯落地文 | **Scripts 是最硬门禁**             |

**裁决**：**互补不冲突**。

| 层            | 处理什么                   | 谁做 |
| ------------- | -------------------------- | ---- |
| Scripts（硬） | **规范类问题**（基础门禁） | 机器 |
| 人工 CR       | **业务语义**（高阶判断）   | 人   |

→ 两者构成 **分层验证**：

1. 先过 Scripts（机器筛规范错）
2. 再过 CR（人聚焦业务对错）

## 第三部分：反模式自查清单（每周复盘）

每周复盘问这 10 个问题：

- [ ] 有没有 AI 借"特殊情况"跳过 Rule？
- [ ] 有没有 PR 漏过 Scripts 检查？
- [ ] dev-map 是否跟代码漂移？
- [ ] 是否有规范停留在文档（没进工具链）？
- [ ] 是否有反复出现的同类错误（说明缺 Rule/Skill）？
- [ ] PM Agent 有没有越界做专业判断？
- [ ] CR 是不是又开始找错字了？
- [ ] 测试数量有没有异常下降？
- [ ] 是否有"动态招聘"型多 Agent 出现？
- [ ] 个人偏好有没有偷偷上升为团队规范？

## 第四部分：反模式补救表（出问题了怎么办）

| 症状                   | 反模式                      | 补救                          |
| ---------------------- | --------------------------- | ----------------------------- |
| AI 老犯同一个错        | Error-Driven 闭环缺失       | 沉淀为 Rule / Skill / Scripts |
| 不同人的代码风格差很大 | 缺人人对齐 / 缺 always Rule | 独裁者拉齐 + 升级关键 Rule    |
| AI 越改越乱            | 缺 SPEC / dev-map           | 补 SPEC、建 dev-map           |
| Scripts 老报警没人理   | Scripts 不够"硬"            | 改为阻塞 PR 合并              |
| CR 速度跟不上          | 缺 Pre-PR                   | 引入 AI 自查 + 标准 PR 文档   |
| 重构永远做不完         | 推倒重来 / 等专项           | 见缝插针                      |
| 大代码库无从下手       | 试图人工遍历                | 专家定向 + AI 穷举扫描        |
| AI 越改文档越多        | 文档不在仓库                | Docs as Code                  |
| 多 Agent 路径不稳      | 去中心化协商                | 改为结构化调度                |
