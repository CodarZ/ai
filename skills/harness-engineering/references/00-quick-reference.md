# 00 - 速查卡（30 秒心法 / 5 误区 / 反模式速查）

> 接到任意 AI Coding 任务，先扫一遍本文。本文是 SKILL.md 的扩展，浓缩高频用的"快速心智"。

---

## 30 秒心法

### 核心隐喻

> **AI Agent = SOTA 模型（野马）+ Harness（缰绳/驾驭系统）= 千里马**

Harness Engineering **不是**"更好的提示词"，**不是**"更强的模型"，而是 **"优化模型运行的环境与机制"**。

### 三条公理（一切方法的根基）

1. **意图转化链有损耗**：需求 → 设计 → 编码 → 测试，每步都有损耗。AI 要降低**全链条**损耗，不只是编码段。
2. **LLM 本质三特征**：上下文决定输出（私有知识为零）/ 概率性（同输入可不同输出）/ 工作记忆有限（Lost in the Middle）。
3. **人类认知是稀缺资源**：AI 时代瓶颈是**人的审查决策带宽**。让人聚焦"判断什么重要"，让 AI 做"穷举扫描"。

### R.E.S.T 四目标（任何 Harness 设计的自评维度）

| 字母             | 含义     | 关键问题                                    |
| ---------------- | -------- | ------------------------------------------- |
| **R**eliability  | 可靠性   | 失败能否恢复？操作是否幂等？行为是否一致？  |
| **E**fficiency   | 效率     | Token/时间/API 预算是否可控？延迟可接受吗？ |
| **S**ecurity     | 安全性   | 权限最小化？沙盒隔离？输入输出过滤？        |
| **T**raceability | 可观测性 | 决策链可追踪？状态可审计？错误可定位？      |

## 5 个必须破除的误区（接到任务先自检）

| 误区                            | 真相                                            | 后果                            |
| ------------------------------- | ----------------------------------------------- | ------------------------------- |
| "代码喂给 AI 就能理解项目"      | **信噪比 + 结构化**决定上下文价值，不是越多越好 | 上下文淹没，AI 抓不到重点       |
| "AI 不适合复杂系统"             | 不是 AI 不够聪明，是关键知识没被结构化提供      | 抱怨 AI 同时放弃 Harness 建设   |
| "AI 提效 = 更快写代码"          | 源头损耗传播最远，AI 应贯穿全链条               | 编码快了但需求/设计错了，返工大 |
| "通过测试就能提交"              | 测试只验行为，不验设计偏差和规范违反            | 跑过测试但架构腐化              |
| "AI 应像人一样独立完成整个任务" | 概率 × 多步 = 错误指数累积，**全自主必失稳**    | 单 Agent 通吃，不可控           |

## 6 大零件速查（Harness 的基本词汇）

| 零件          | 回答什么问题       | 软/硬约束      | 何时引入        |
| ------------- | ------------------ | -------------- | --------------- |
| **Rule**      | 什么事绝对不能乱来 | 软（AI 会忘）  | 第 2 步         |
| **Skill**     | 这件事具体怎么做   | 半软           | 第 3 步         |
| **Sub Agent** | 谁负责哪段         | 角色契约       | 单 Agent 失稳后 |
| **Workflow**  | 什么顺序接力       | 流程           | 多 Agent 复杂后 |
| **Scripts**   | 谁判断做对没       | **硬（最硬）** | 第 1 步后立刻补 |
| **MCP**       | 怎么接外部系统     | 接口           | 想闭环往外推时  |

**关键认知**：

- Rule 只能做**原则约束**，不能做**流程执行** → 流程必须脚本化
- Scripts 是 Harness 里**最硬**的东西，"你说做完了没用，得跑过我这关才算"

## 反模式速查（21 条最常犯）

### 概念误用

- ❌ "Harness = 更好的 prompt" → Harness 是环境与机制
- ❌ "Memory 是核心组件" → 团队级 Harness 里**对话级 Memory 应靠边**
- ❌ "Rule 能解决所有约束" → Rule 是软约束，会被"解释性执行"
- ❌ "把整仓喂给 AI 它就懂" → 信噪比决定价值

### 流程错误

- ❌ 下游自己改上游文档 → 必须打回上游
- ❌ PM Agent 越界做专业判断 → PM 只做路由
- ❌ 一个 Agent 干到底 → 必然失稳
- ❌ 去中心化自由协商（AutoGen GroupChat 类） → 路径不稳定
- ❌ 流程靠 PM 长文记忆 → 必须抽成可校验资产

### 验证缺失

- ❌ "通过测试就能提交" → 测试只验行为
- ❌ "我觉得做完了" → 必须 Scripts 判定
- ❌ AI 自动生成全部测试用例 → 漏隐性高危，堆无价值边缘 case
- ❌ "这错误本来就有" → 用基线对比堵住

### 知识管理

- ❌ 项目级知识库做百科全书 → 应该是索引
- ❌ 个人偏好上升为团队规范 → 应纳入仓库资产
- ❌ 规范停留在文档不进工具链 → 一纸空文

### 起步策略

- ❌ 第一天就全搭齐 → 应渐进演化
- ❌ 推倒重来式重构 → 见缝插针
- ❌ 申请专项排期重构 → 拆解能力决定一切
- ❌ 代码库混乱时直接放 AI 写 → AI 不纠偏反而放大差异
- ❌ 配 AI Rule 就万事大吉 → 真正瓶颈在人的共识
- ❌ 跳过"人人对齐"直接配 Rule → 顺序错了 Rule 是一纸空文

## 核心金句（建立直觉用）

> **"真正贵的不是 token，真正贵的是失控。"** —— 腾讯/白家杰

> **"AI 不是助手，AI 更像是一支执行力极强但必须被制度化管理的团队。"** —— 腾讯/白家杰

> **"事实像档案，例子像故事，规矩像手册——手册应进仓库，别只靠聊天记忆。"** —— Memory 三句话

> **"1 个独裁者好过 10 个民主者。"** —— 美团

> **"AI Coding 不会自动收敛复杂度，没有规范约束，AI 只会成倍放大混乱。"** —— 美团

> **"经验的价值正在从'能看全'转移到'能判断什么重要'。"** —— 美团

> **"重构不需要排期，需要拆解能力。"** —— 美团

> **"通过有序的约束来驾驭无序的算力。"** —— TRAE.ai

> **"必须将 LLM 严格视为无状态的计算单元（CPU）。"** —— TRAE.ai

> **"不要贪大，不要一步到位，先从你最反复、最痛的那个问题开始。"** —— 腾讯/白家杰

> **"真正专业的 Harness，不应该越来越像'我和我的 AI 的默契'，而应该越来越像'任何一个人拿到这个工程，都能顺着这套系统做对事情'。"** —— 腾讯/白家杰
