# 05 - 知识库（dev-map / 任务看板 / Memory 定位）

> 本文回答："项目知识怎么组织？AI 怎么找到该看的东西？" 项目级知识库是"索引"而非"百科全书"。

---

## 1. 核心观点

> **项目级上下文是"索引"而非"百科全书"。**

**反模式**：把整仓说明书塞进 AI 的 context
**正模式**：建立索引系统，AI 按需查阅

---

## 2. dev-map（开发导航地图）

> 来源：腾讯 / 白家杰

### 2.1 定义

dev-map 是项目的开发导航地图，回答"改代码前，我该看哪里"。

| 项           | 内容                                       |
| ------------ | ------------------------------------------ |
| **维护者**   | **开发 Agent**（谁动代码谁改地图）⭐       |
| **内容**     | 功能落点、影响范围、标准写法、模块联动链路 |
| **规则**     | **"改代码之前，先查 dev-map"**             |
| **仓库大时** | 拆成几本薄册子，封面写清管哪一片           |

### 2.2 内容结构示例

```markdown
# dev-map / 用户认证模块

## 功能落点

- 登录入口: src/auth/LoginController.cs
- Token 生成: src/auth/TokenService.cs
- 中间件: src/middleware/AuthMiddleware.cs

## 影响范围

- 修改 TokenService → 影响所有 API 调用
- 修改 LoginController → 影响登录页 + 后台管理页

## 标准写法

- 新增认证方式 → 在 IAuthProvider 接口下加实现
- 不要直接读写 last_state.json，用 SessionManager

## 模块联动链路

登录 → TokenService.GenerateToken
→ SessionManager.Create
→ 审计日志（异步）
```

### 2.3 维护纪律

**关键规则**：**谁动代码谁改地图**。

为什么不让 PM 改？

- PM 不在第一线，地图会跟代码漂移
- 开发 Agent 改代码时**必须**更新对应 dev-map 章节
- 这是 Scripts 可以检查的项（diff 涉及 src/auth → 必须 diff docs/dev-map/auth.md）

### 2.4 大仓库的处理

不要做成一个 10MB 的大文件。**拆分原则**：

- 按模块拆（auth/、payment/、reporting/）
- 每本薄册子有"封面"说明管哪一片
- 总入口 INDEX.md 指引去哪本

## 3. 任务看板（项目态势板）

> 来源：腾讯 / 白家杰

### 3.1 定义

任务看板是项目的态势板，回答"项目当前在做什么"。

| 项         | 内容                                                 |
| ---------- | ---------------------------------------------------- |
| **维护者** | **PM**                                               |
| **内容**   | 当前任务、各阶段进度、文档目录、已完成任务的交付结论 |
| **作用**   | 防止"新方案把旧方案冲掉"、避免重复造轮子             |

### 3.2 看板内容示例

```markdown
# 项目任务看板

## 进行中

- [#127] 用户认证重构 (方案设计阶段) → docs/specs/127.md
- [#128] 性能优化批次 1 (开发阶段) → docs/specs/128.md

## 已完成（最近 10 个）

- [#125] 修复 SVN 冲突自动处理 → 交付结论: 引入 ConflictResolver 类
- [#124] dev-map 拆分 → 交付结论: 按模块拆成 5 个 sub-map
  ...

## 阻塞中

- [#126] 多语言支持 → 阻塞原因: 等待产品确认翻译方案

## 文档目录

- docs/specs/ - 各任务的 SPEC
- docs/dev-map/ - 开发导航地图
- docs/process/ - 流程定义与契约
```

### 3.3 为什么需要

**问题场景**：

- 工程师 A 做了"认证重构"方案 #127
- 工程师 B 不知情，又开了 #135 做类似的事
- 两个方案冲突

**任务看板**：让所有人/Agent 都能快速看到"当前在做什么 + 已经做过什么"。

## 4. 项目级索引的其他形式

> 来源：腾讯 / 白家杰

dev-map 和任务看板是核心，但还有其他备选索引形式：

| 形式                 | 适用场景                         |
| -------------------- | -------------------------------- |
| 人工导航文档         | 小项目，手写够用                 |
| 任务总览             | 中型项目，需要全局视图           |
| 工具自动生成的地图   | 依赖图、调用图（自动化但粗粒度） |
| 编辑器跳转           | LSP 提供，细粒度但缺业务语义     |
| 检索（向量检索兜底） | 大仓库，关键词找不到时           |
| README / CODEOWNERS  | 模块入口                         |
| 独立 Wiki            | 跨项目知识                       |

**组合使用**，不是替代关系。

## 5. Memory 的定位（最易混淆的部分）

> 来源：腾讯 / 白家杰（反主流观点） + TRAE.ai（架构观点）

### 5.1 区分两种 "Memory"

合并 4 篇文章时最大的潜在冲突点。**必须分清**：

| 类型              | 含义                                            | 文章中谁在说              |
| ----------------- | ----------------------------------------------- | ------------------------- |
| **对话级 Memory** | Claude/ChatGPT 自动记忆、用户口头约定、聊天历史 | 腾讯落地文（应靠边）      |
| **系统级 Memory** | 外部存储的 RAG 知识库、向量数据库、状态存储     | TRAE 文（三层架构必须有） |

两者**不是同一个东西**。讨论 Memory 时务必先确认是哪种。

### 5.2 对话级 Memory 的三句话定位

> **"事实像档案，例子像故事，规矩像手册——手册应进仓库，别只靠聊天记忆。"** —— 腾讯/白家杰

| 类型                 | 像什么   | 该落在哪                     |
| -------------------- | -------- | ---------------------------- |
| 结论、边界、任务索引 | **档案** | SPEC、任务看板、dev-map      |
| 成功/失败样例        | **故事** | 测试用例、门禁脚本、评审清单 |
| 反复踩的坑、红线     | **手册** | **优先 Scripts，其次 Rule**  |

### 5.3 对话级 Memory 适用场景

| 场景                     | Memory 地位                                        |
| ------------------------ | -------------------------------------------------- |
| 个人工具/小型项目        | **性价比高**，省重复话                             |
| 多人、标准、长期维护工程 | **必须以仓库资产 + 门禁为主干**，Memory 只做润滑剂 |

### 5.4 为什么团队级 Harness 里对话级 Memory 应靠边

理由：

- **矛盾**：同人 Memory 可能矛盾（"上次说要 A"、"上上次说要 B"）
- **不可见**：交接 / 审计 / CI 看不见 Memory
- **隐性规范**："个人风格"易上升为隐性团队规范，损害标准化
- **不可版本化**：Memory 通常不在仓库

### 5.5 系统级 Memory（三层架构）

> 来源：TRAE.ai

这个 Memory 指**外部存储的知识系统**，是 Harness 的必要组件。

| 层           | 用途                | 实现             |
| ------------ | ------------------- | ---------------- |
| **短期**     | 当前会话/任务上下文 | 进程内状态       |
| **长期**     | 历史交互/用户偏好   | 持久化存储       |
| **知识检索** | 项目知识库          | RAG / 向量数据库 |

**关键纪律**：注入边界（Injection Boundary）至关重要：

- RAG 结果在 Prompt 的 **位置**（前/中/后）会影响效果
- 避免 **Lost in the Middle** 问题
- 通常重要信息放 Prompt 头部或尾部

## 6. Knowledge as Code（知识即代码）

### 6.1 核心思想

把以下内容**编码为版本化的 Rule/Skill**：

- 编码规范
- 设计原则
- 业务约定
- 反模式清单
- 项目特殊规则

### 6.2 好处

- AI 成为**知识分发载体**
- 每个团队成员协作时**自动获得**团队最佳实践加持
- 知识可版本化、可审计、可演进

### 6.3 反模式对比

| 反模式                   | 正模式                           |
| ------------------------ | -------------------------------- |
| 知识在老员工脑子里       | 知识在 Rule/Skill 文件里         |
| 知识在 Confluence/Notion | 知识在仓库 + 版本化              |
| 知识写一次永不更新       | 知识有 Error-Driven 闭环更新机制 |

---

## 7. 反模式（知识库层）

| 反模式                            | 真相                   |
| --------------------------------- | ---------------------- |
| 项目级知识库做成百科全书          | 应该是**索引**         |
| 个人偏好上升为团队规范            | 应纳入仓库资产         |
| 规范停留在文档（不进工具链）      | **一纸空文**           |
| 同一规则散落 Rule / Memory / 聊天 | 三句话定位法明确归属   |
| dev-map 由 PM 维护                | 应由**改代码的人**维护 |
